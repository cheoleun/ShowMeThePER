# ShowMeThePER 요구사항

## 개요

ShowMeThePER는 KOSPI와 KOSDAQ에 상장된 한국 기업의 과거 재무제표 데이터를 수집하고 보여주는 프로그램이다.

제품의 핵심 관심사는 단순히 재무제표 원문 숫자를 나열하는 것이 아니라, 최근 성장률 흐름을 중심으로 기업을 비교하고 필터링할 수 있게 하는 것이다. 특히 매출, 영업이익, 순이익의 성장률을 기본 분석 축으로 삼는다.

## 핵심 데이터 요구사항

- KOSPI, KOSDAQ 상장기업의 과거 재무제표 데이터를 수집한다.
- 초기 버전은 웹에서 직접 수집하거나 공개 Open API를 사용해 데이터를 가져온다.
- 수집 데이터가 많아지면 반복 조회 비용을 줄이기 위해 DB에 저장하고 재사용하는 구조로 전환한다.
- 기본 화면에서 최근 10년 정보를 보여줄 수 있을 만큼의 과거 데이터를 확보한다.

## 기본 성장률 화면

기본 화면은 최근 10년 기준 성장률 정보를 숫자 데이터와 차트로 함께 보여준다.

기본으로 제공할 성장률 지표는 다음과 같다.

- 분기별 YoY 매출 증가율
- 분기별 YoY 영업이익 증가율
- 분기별 YoY 순이익 증가율
- 최근 4분기 합산 기준의 분기별 YoY 매출 증가율
- 최근 4분기 합산 기준의 분기별 YoY 영업이익 증가율
- 최근 4분기 합산 기준의 분기별 YoY 순이익 증가율
- 연간 매출 증가율
- 연간 영업이익 증가율
- 연간 순이익 증가율

“최근 4분기 합산”은 현재 분기를 포함한 최근 4개 분기 합산값을 전년 동일 4개 분기 합산값과 비교한다는 의미다.

## 기업 필터링과 순위

기업 필터링의 기본 필터 그룹은 성장률 중심으로 제공한다. PER, PBR, ROE는 기본 필터가 아닌 선택 필터와 순위 지표로 제공한다.

성장률 필터는 매출, 영업이익, 순이익을 각각 독립적으로 선택할 수 있어야 한다. 사용자는 어떤 성장률 지표를 필터 조건으로 적용할지 직접 고를 수 있어야 한다.

성장률 기본 필터 프리셋은 다음과 같다.

- 기간: 최근 3년
- 기준값: 20% 이상
- 판정 방식: 선택된 기간 내 모든 값이 기준값 이상이어야 통과

구현 관점에서는 선택된 성장률 시계열의 최근 3년 구간에서 최소 성장률이 기준값 이상이면 통과로 본다. 즉, 조건은 `minimum >= threshold`로 표현한다.

분기별 YoY 성장률에서 최근 3년은 최근 12개 분기를 의미한다. 연간 YoY 성장률에서 최근 3년은 최근 3개 회계연도를 의미한다.

성장률 필터의 기준 시계열은 분기별 YoY 성장률과 연간 YoY 성장률을 모두 지원한다.

순위 화면 또는 순위 목록은 다음 지표를 제공한다.

- PER
- PBR
- ROE
- 매출 성장률
- 영업이익 성장률
- 순이익 성장률

## 추후 설계가 필요한 계산 규칙

전년 동기 값이 0 이하이거나 결측인 경우 성장률을 어떻게 계산하거나 제외할지는 구현 전에 별도 설계 항목으로 확정한다.

재무제표 데이터 소스, 기업 식별자, DB 모델, 차트와 테이블 UI, 필터 API, 순위 API도 구현 전에 별도로 설계한다.

## 개발 원칙

- 모든 작업은 GitHub Pull Request 단위로 올린다.
- 기능 구현 시 가능한 한 테스트 코드를 함께 작성한다.
- 향후 구현에서는 데이터 수집, 성장률 계산, 필터링, 순위 산출, 차트와 테이블 표시 로직에 대한 테스트를 작성한다.

## GitHub 작업 흐름

사용자가 “구현해줘”, “진행해줘”, “PR 올려줘”처럼 작업 의사를 명확히 밝힌 경우에는 별도 승인 질문 없이 다음 흐름을 한 번에 실행한다.

- 별도 브랜치를 생성한다.
- 요청된 변경을 구현한다.
- 필요한 검증이나 테스트를 실행한다.
- 변경사항을 커밋한다.
- GitHub에 브랜치를 push한다.
- draft Pull Request를 생성한다.
- 작업 결과, PR 링크, 검증 결과를 공유한다.

다음 경우에는 안전을 위해 진행 전에 사용자 확인을 요청할 수 있다.

- 작업 트리에 사용자가 만든 것으로 보이는 변경사항이 섞여 있어 PR에 포함할 파일 범위가 불명확한 경우
- `git reset`, 강제 push, 대량 파일 삭제처럼 파괴적이거나 되돌리기 어려운 작업이 필요한 경우
- 권한, 네트워크, 인증 문제로 시스템 승인이 필요한 경우
- Pull Request merge, ready for review 전환처럼 게시 이후 단계의 결정이 필요한 경우
